ere’s an at-a-glance inventory of where the application currently persists its data.

Firestore (cloud persistence)

Saved in Firestore

moves collection
– Top-level move documents (moveCode, ownerId, participants, timestamps)
moves/{moveId}/boxes sub-collection (all box records)
moves/{moveId}/owners sub-collection (personal owners; communal rooms are hard-coded)
presence collection (user presence per move)
users collection (user profile created/updated during authentication)
Not yet saved in Firestore

Budget feature data (budgets, categories, expenses, receipts)
Theme preference
Local application settings (defaultBatchPrintCount, currentMoveId)
New-user flags (isNewUser, userFullName)
Any other UI-only state (filters, modal visibility, etc.)
Local persistence (browser localStorage / sessionStorage)

Saved locally

budget-tracker-state (usePersistentReducer)
– Expenses, categories, total budget, move type
THEME_STORAGE_KEY (stored by useTheme)
– Current light/dark theme selection
SETTINGS_LOCAL_STORAGE_KEY (settingsService)
– AppSettings object: defaultBatchPrintCount, currentMoveId, etc.
SessionStorage flags (set in Auth & App start-up)
– isNewUser (boolean string)
– userFullName (cached full name for welcome flow)
Not stored locally

Boxes, owners, moves, presence, and users data (all live exclusively in Firestore)
Uploaded receipt images once the app is refreshed (they exist only inside the persisted budget state, not separately cached)
Any server-derived data that’s streamed directly from Firestore listeners
